  <action>
The server GET /events/{id}/check-in currently returns List&lt;CheckInRow&gt; (flat: eventId, userId, status, note, setBy, setAt, previousStatus, previousSetBy). The client deserializes as List&lt;CheckInEntry&gt; which has a completely different shape: {userId, userName, userAvatar, response: AttendanceResponse?, record: AttendanceRecord?}.

Fix approach -- change the server to return the joined shape:

1. In `AttendanceRepository.kt`, add a new server-side DTO:
```kotlin
@Serializable
data class CheckInEntryResponse(
    val userId: String,
    val userName: String,
    val userAvatar: String? = null,
    val response: AttendanceResponseDto?,
    val record: AttendanceRecordDto?
)

@Serializable
data class AttendanceResponseDto(
    val eventId: String,
    val userId: String,
    val status: String,
    val reason: String? = null,
    val abwesenheitRuleId: String? = null,
    val manualOverride: Boolean = false,
    val respondedAt: Instant? = null,
    val updatedAt: Instant
)

@Serializable
data class AttendanceRecordDto(
    val eventId: String,
    val userId: String,
    val status: String,
    val note: String? = null,
    val setBy: String,
    val setAt: Instant,
    val previousStatus: String? = null,
    val previousSetBy: String? = null
)
```
Use kotlinx.serialization @Serializable + kotlinx.datetime.Instant (NOT java.time). Import the kotlinx.serialization.Serializable and add `import kotlinx.datetime.Instant as KInstant` if needed to disambiguate from java.time.Instant. Alternatively, put these DTOs in a new file or in CheckInRoutes.kt as private/internal to avoid polluting the domain repo interface.

Simpler approach: Add a new method `getCheckInEntries(eventId: UUID): List<CheckInEntryResponse>` to the repository interface, or build the joined response directly in the route handler.

2. In `AttendanceRepositoryImpl.kt`, add `getCheckInEntries` that:
   - JOINs `team_roles` (to get event's team members), `users` (for userName, avatar), `attendance_responses` (LEFT JOIN for response), `attendance_records` (LEFT JOIN for record)
   - For each team member of the event: builds a CheckInEntryResponse with nested response + record objects
   - Query: Get all team members for the event's teams via events_teams JOIN team_roles JOIN users, then LEFT JOIN attendance_responses and attendance_records on (eventId, userId)
   - Convert java.time.Instant to kotlinx.datetime.Instant for DTO serialization using `toKotlinInstant()` from `kotlinx.datetime`

3. In `CheckInRoutes.kt`, change the GET handler:
```kotlin
get("/events/{id}/check-in") {
    val eventId = UUID.fromString(call.parameters["id"])
    val entries = attendanceRepo.getCheckInEntries(eventId)
    call.respond(entries)
}
```

IMPORTANT: The JSON field names in the server response MUST exactly match the shared CheckInEntry fields: `userId` (String), `userName` (String), `userAvatar` (String?), `response` (object with eventId/userId/status/reason/abwesenheitRuleId/manualOverride/respondedAt/updatedAt), `record` (object with eventId/userId/status/note/setBy/setAt/previousStatus/previousSetBy). All UUID fields in DTOs must be serialized as String. All Instant fields must use kotlinx.datetime.Instant (not java.time).

Note: The existing `getCheckIn` method and `CheckInRow` can remain for internal use (e.g., upsertCheckIn still returns CheckInRow). Only the GET route response changes.

V8 migration made team_roles.user_id nullable with SET NULL on delete. But TeamRolesTable Exposed object still declares it non-nullable with CASCADE. This causes NPE when reading rows where user_id IS NULL after member removal.

In `TeamsTable.kt`, change TeamRolesTable.userId from:
```kotlin
val userId = uuid("user_id").references(UsersTable.id, onDelete = ReferenceOption.CASCADE)
```
to:
```kotlin
val userId = uuid("user_id").references(UsersTable.id, onDelete = ReferenceOption.SET_NULL).nullable()
```

Also update the unique index since userId is now nullable -- Exposed handles nullable columns in unique indices, but verify the `uniqueIndex` call still compiles. The index declaration `uniqueIndex("idx_team_roles_user_team_role", userId, teamId, role)` should still work with a nullable column.

Then grep for all usages of `TeamRolesTable.userId` across the server codebase. Any code that reads `row[TeamRolesTable.userId]` previously returned `UUID` but now returns `UUID?`. Fix all call sites to handle the nullable:
- In queries that filter `WHERE user_id = X`, these are fine (comparing nullable column to value works)
- In row mapping functions that read `row[TeamRolesTable.userId]`, these need `?.` or null checks
- Specifically check TeamRepositoryImpl.kt for row mapping functions

Common pattern: `row[TeamRolesTable.userId]` becomes `row[TeamRolesTable.userId] ?: return@map null` or similar, then filter nulls from the list.
